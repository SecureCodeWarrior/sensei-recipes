Not having a custom error page for the HTTP 500 'Internal Server Error' could lead to information leakage.


Default error pages often contain full stack traces of the errors that are encountered. This discloses information about the inner workings of the application that could be abused by attackers.


Before 
<web-app>
...
</web-app>

<web-app>
    <error-page>
        <error-code>400</error-code>
    </error-page>
</web-app>
After 
<web-app>
    <error-page>
        <error-code>500</error-code>
    </error-page>
</web-app>

<web-app>
    <error-page>
        <error-code>400</error-code>
    </error-page>
    <error-page>
        <error-code>500</error-code>
    </error-page>
</web-app>
